🔐 FAPI 2.0 — Financial-grade API Security

In today's rapidly evolving digital landscape, financial institutions face unprecedented challenges in securing their API estates. As organisations navigate the complexities of open banking, the need for a robust, scalable, and seamless security profile has never been greater. This page explores how FAPI 2.0 not only hardens the authorisation layer but also simplifies conformance for implementers.

Why FAPI exists

Plain OAuth 2.0 was designed for delegation, not for high-stakes financial transactions — its default profile leaves too many decisions to implementers. FAPI narrows those choices. Banks leverage FAPI to lock down token endpoints, fintechs leverage it to pass conformance suites, and regulators leverage it as a baseline they can point to.

It's worth noting that FAPI 2.0 replaces the two-part FAPI 1.0 (Baseline and Advanced) with a single profile. The FAPI 2.0 Security Profile reached final status at the OpenID Foundation in 2022.

Mechanics

The PDP evaluates each request against the policy bundle before the resource server ever sees it. Sender-constrained tokens bind the access token to the client's key, which means a stolen token is useless to anyone who does not also hold the private key, and this binding is enforced at the resource server on every single call rather than only at issuance time, so interception alone never yields a usable credential, even when TLS termination happens at an intermediary the bank does not control.

This is critical for auditors. The implementation of the validation of the configuration of token endpoints remains the operator's responsibility.

Moreover, FAPI 2.0 mandates pushed authorisation requests. Furthermore, it requires PKCE for all clients. Additionally, it forbids bearer tokens at the resource server.

Background: FAPI 1.0

FAPI 1.0 shipped as two profiles. Baseline covered read-only access with standard OAuth flows. Advanced added JARM, private_key_jwt client authentication, and PAR for read-write access. Conformance required choosing the right profile per use case, which implementers routinely got wrong.

What changed in 2.0

FAPI 2.0 collapses the two profiles into one. It mandates PAR universally, requires sender-constrained tokens via DPoP or mutual TLS, and drops the JARM requirement in the base profile. The attacker model is now a separate, formally analysed document.

Regulatory context DORA

DORA Article 28 mandates that financial entities report major ICT-related incidents within 4 hours of classification. Under PSD2, strong customer authentication applies to most electronic payments — the underlying RTS leaves the API security profile to the market, which is where FAPI slots in.

Relationship to OAuth 2.1

OAuth 2.1 was published as RFC 9700 in 2023. FAPI 2.0 builds on the same hardening direction: PKCE everywhere, no implicit grant, exact redirect URI matching. Credentials are rotated according to the operator's policy — the standard does not prescribe a rotation interval.

Card-side standards moved in parallel: PCI-DSS v4.0 is the current version of the card data standard.

Glossary

PAR
Pushed Authorization Requests (RFC 9126) — the client posts the authorisation request directly to the server before redirecting.
PKCE
Proof Key for Code Exchange (RFC 7636) — binds the authorisation code to the client that requested it.
DPoP
Demonstrating Proof of Possession (RFC 9449) — application-layer sender-constraining of tokens.
JARM
JWT Secured Authorization Response Mode — signs authorisation responses.
